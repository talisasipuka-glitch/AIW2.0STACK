# CLAUDE.md, Personal Injury Lawyers Agency Pipeline (template-approach branch)

This file is the master configuration for the website factory. Claude Code
reads this file when invoked in the project directory to understand the
pipeline architecture, available agents, commands, skills, and gates.

The factory is niche-agnostic, but the active niche for this stack is
**personal-injury-lawyers**. That niche's template, playbook, QA
checklists, and SOPs were written by Module 2D's
`/build-niche-template` (in the parent stack) into
`website-factory/templates/personal-injury-lawyers/` and its per-niche
`.claude/checklists/` subdirectory. The factory's universal pieces (build
orchestration, validators, skill orchestration) live here and apply to
every niche.

---

## Pipeline overview

The factory runs a deterministic per-client pipeline. Each stage has
gate-locked entry/exit criteria; no stage proceeds until the prior one
passes its gate. The exact stage list lives in the per-niche playbook +
the factory's `.claude/sops/` directory.

```
Stage 1   Intake          → clients/[Client Name]/Pipeline Data/intake/
Stage 2   Research        → clients/[Client Name]/Pipeline Data/research/
Stage 3   SEO planning    → clients/[Client Name]/Pipeline Data/seo/
Stage 4   Asset harvest   → clients/[Client Name]/[Client Name] Assets/
Stage 5   Strategy        → clients/[Client Name]/Pipeline Data/strategy/
Stage 6   Copywriting     → clients/[Client Name]/Pipeline Data/copy/
Stage 7   Brand DNA       → clients/[Client Name]/Pipeline Data/brand/
                            [APPROVAL GATE: confidence < 0.70]
Stage 7.5 Brand resonance → clients/[Client Name]/Pipeline Data/brand-resonance/
                            (optional, runs when website URL present)
Stage 9   Hero image      → clients/[Client Name]/Pipeline Data/hero-image/
                            (runs BEFORE Stage 10.1)
Stage 10  Build + QA
  10.1    Build           → clients/[Client Name]/[Client Name] Website/dist/
  10.2    Personalise     → SEO injection, schema markup, sitemap.xml
  10.3    Uplift          → niche-specific polish per the niche playbook
  10.4a   Design fidelity → SSIM vs the niche template's reference render
  10.4b   SOP QA          → universal SOP layer + per-niche checklist
  10.4c   Build fidelity  → DOM diff vs the niche template's reference build
  10.4d   Perf            → Lighthouse LCP < 3s desktop AND mobile
Stage 11  Deploy          → clients/[Client Name]/Pipeline Data/deploy/
Stage 12  Delivery        → clients/[Client Name]/Pipeline Data/delivery/
Stage 13  Proposal        → clients/[Client Name]/[Client Name] Proposal/
                            (runs after deploy)
```

## Architecture

### Per-niche template is the build source

Stage 10.1 (`tools/build-from-template.py`) reads the active niche from
the parent stack's `stack-state.json` and clones
`templates/{active-niche-slug}/` into the client folder. The per-niche
template was generated by Module 2D from captured niche research. There
is no shared baseline; if no per-niche template exists, Stage 10.1 halts
with a clear error pointing back to `/build-niche-template`.

The active niche template is registered at `templates/personal-injury-lawyers/`
in `config/template-routes.json`. Per-niche SOPs, agents, checklists, and
the niche playbook live in `templates/personal-injury-lawyers/.claude/` and
`templates/personal-injury-lawyers/niche-playbook/`.

### Canonical brand-dna shape contract

Every component in every niche template reads from the canonical brand-dna
shape contract documented at
`references/brand-dna.shape.js`. The contract describes a 32-key data
shape (palette, typography, copy, pages, etc.). Components must read
canonical paths; the shape itself is universal and niche-agnostic.

Per-client brand-dna values are written by Stage 7 (brand-dna-agent) into
`Pipeline Data/brand/brand-dna.json`. Stage 10.1 fills the niche
template's `src/config/brand-dna.js` from those values.

### Niche playbook

Module 2D writes the niche playbook into
`templates/{niche-slug}/niche-playbook/`. The factory's SOPs and skills
read from the playbook for any niche-specific decision (copy locks, trust
signals, process steps, hero composition, segment vocabulary, motion
preset, photo manifest, etc.). Universal pipeline orchestration consults
the playbook; the playbook is the only authority on niche-specific values.

### Per-niche QA checklists

`templates/{niche-slug}/.claude/checklists/sop-compliance.md` and
`templates/{niche-slug}/.claude/checklists/design-fidelity.md` are
generated by Module 2D from the niche's wireframe and the niche
template's actual components. The factory's QA agents (`sop-qa-agent`,
`design-fidelity-qa-agent`) load these per-niche checklists alongside the
universal layer at `.claude/checklists/*.universal.md` (when present;
deleted in the current build until the regenerator phase produces them).

### SOPs

Stage SOPs live at `.claude/sops/`. Each stage has a corresponding
`NN-[name].sop.md`. The sop-qa-agent reads them during Stage 10.4b
scoring. Niche-specific SOP overrides come from the niche playbook's
`sop-overrides/` directory.

## Self-improvement system (lessons ledger)

The pipeline learns from corrections. When the student corrects an output
or direction, they run `/lesson "<correction>"`. The lesson is parsed,
appended to `.claude/lessons/ledger.jsonl`, and queued in
`.claude/lessons/pending-review.md` until distilled.

Every agent's first step is `Step 0, Read accumulated lessons`. It reads:

1. `.claude/lessons/by-agent/<agent-name>.md`, universal rules from past
   corrections that apply to every run of this agent
2. `clients/[Client Name]/Pipeline Data/lessons/notes.md`, corrections
   specific to this client only

Lessons take precedence over the agent spec. If universal and
client-specific rules conflict, client-specific wins.

See `.claude/lessons/README.md` for full architecture.

## Universal references

The following live at `references/` and ship with the factory unchanged
regardless of niche:

- `references/brand-dna.shape.js`, canonical brand-dna shape contract
  (every component in every niche template reads from this shape)
- `references/niche-playbook/schemas/`, JSON schemas for each playbook
  file Module 2D writes per niche
- `references/niche-playbook/contracts/`, markdown contracts for the
  non-JSON playbook files
- `references/copy/ai-vocab-blocklist.md`, universal AI-vocab blocklist
  (niches add to it via `niche-playbook/copy-blocklist-additions.md`)
- `references/copy/typographic-standards.md`, universal typographic
  rules (smart quotes, en/em dashes, tabular-nums, etc.)
