# 10.4c - Build Fidelity

Implements: Stage 10.4c.

## Purpose

Run a DOM structural diff between the client build and the niche
template's reference build, confirming the 10-page sitemap, including all
applicable case-type routes, matches structurally.

## Inputs (per-client)

```
clients/[Client Name]/[Client Name] Website/dist/
templates/personal-injury-lawyers/ (reference build)
research/02-niche-research/personal-injury-lawyers/09-sitemap.json
```

## Procedure

1. Build the reference template (if not already built) for comparison.
2. Walk both DOM trees for: Home, About, Practice Areas, Testimonials, FAQ,
   Contact, Blog, and every case-type route present in the client's
   `09-sitemap.json`-derived route list (up to 7).
3. For each route, diff: element hierarchy, component boundaries (Header,
   Hero, ProcessSteps, StatsBar, PressBand, WhyChooseUs, PracticeAreasGrid,
   FounderStory, Testimonials, FAQSection, CTABand, Footer for Home; the
   CaseTypeHero/RiskRemovalBand/StatsBar/CaseTypeExplainer/FAQSection/
   CTABand+LeadForm sequence for case-type pages), ARIA roles, and the
   presence of expected `data-` or component identifier attributes.
4. Confirm every `LeadForm` instance has exactly 4 `<input>` fields in the
   DOM (name, phone, email, case description), matching the SOP QA check
   from Stage 10.4b at the structural level.
5. Confirm the header's sticky CTA element (CALL NOW link) and, where
   applicable, the "OPEN 24/7" badge are present in the DOM with the
   expected sticky positioning class/attribute.
6. Confirm `BreadcrumbList`-supporting DOM structure (e.g. a breadcrumb nav
   element) is present on every case-type page.
7. Report any structural divergence: missing component, extra/duplicated
   component, wrong nesting order, missing ARIA roles.

## Pass criteria

- DOM structure for every page in the client's confirmed sitemap matches
  the reference template's structure for the equivalent page type.
- Every confirmed case-type route's DOM matches the CaseTypeLandingPage
  reference structure (CaseTypeHero, RiskRemovalBand, StatsBar,
  CaseTypeExplainer, FAQSection, CTABand+LeadForm, in that order).
- Every LeadForm instance has exactly 4 input fields at the DOM level.
- Sticky header CTA element present with correct positioning.
- Breadcrumb structure present on every case-type page.

## When this stage halts

- A confirmed case-type route's DOM is missing one or more of the 6
  CaseTypeLandingPage sections, or sections appear out of order.
- A LeadForm instance has a DOM-level field count other than 4.
- The homepage section order in the DOM does not match
  `09-wireframe.md`'s Hero -> ProcessSteps -> StatsBar -> PressBand ->
  WhyChooseUs -> PracticeAreasGrid -> FounderStory -> Testimonials ->
  FAQSection -> CTABand sequence.

## Niche-specific notes

For this niche, the following pattern applies (derived from
`research/02-niche-research/personal-injury-lawyers/09-template-spec.md` +
the niche playbook):

This stage's most niche-specific check is the case-type route set: the
client's DOM should contain a CaseTypeLandingPage instance for every case
type present in `brandDNA.services[]`, structurally identical to the
template's reference CaseTypeLandingPage component, just with different
content. A client covering 5 of the 7 case types should have exactly 5
structurally-matching case-type routes, not 7 routes with 2 empty
placeholders and not 5 routes with divergent structure.
